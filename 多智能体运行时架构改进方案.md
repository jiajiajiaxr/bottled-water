# 多智能体运行时架构改进方案

## 概述

本文档基于现有代码的系统性分析，提出从"单线程批处理脚本"向"自持协程多智能体系统"演进的完整改进方案。核心思想：**框架负责流程闭环与资源守护，Agent 负责自主推理与状态自持，EventBus 负责全过程可观测。**

---

## 一、接口层改进

### 1.1 修复 ToolExecutor 接口断裂

**问题**：接口定义 `execute(tool_name, parameters)`，实现类签名却是 `execute(tool_call: ToolCall)`，`AgentLoop` 按接口调用时直接崩溃。

**措施**：

- 统一接口签名：`execute(tool_name: str, parameters: Dict[str, Any]) -> ToolResult`
- 移除 `list_tools()` 方法，该职责移交 `ToolRegistry`
- `ToolExecutor` 仅保留"执行抽象"，不做工具发现

### 1.2 EventSink 职责上提

**问题**：`EventSink` 被 `Orchestrator` 持有但从未使用，且位置错误，导致事件分发与调度逻辑耦合。

**措施**：

- 删除 `Orchestrator` 对 `EventSink` 的依赖
- `EventSink` 上提至 `Session` 层，由 `EventDispatcher` 统一管理
- 支持多 Sink 并发消费：SSE、日志系统、持久化存储、消息队列

```
Session (EventDispatcher)
├── SSESink → 前端实时流
├── LoggerSink → 日志系统
├── DBSink → 持久化
└── KafkaSink → 消息队列（可选）
```

### 1.3 执行器职责精简

**措施**：

- `ToolExecutor` 只回答"怎么跑"（执行抽象）
- `ToolRegistry` 回答"有哪些"（Schema 管理）
- `AgentConfig.tools` 回答"谁能看见"（权限过滤）

## 二、数据层改进

### 2.1 让 AgentContext 真正被消费

**问题**：`AgentContext` 虽然存在，但 `AgentLoop` 完全不消费。每轮 Agent 执行都是失忆重启，思维链、工具调用历史、中间结果全部丢失。

**措施**：

- `AgentLoop` 构造 LLM 消息时，将 `AgentContext.frames` 按角色注入：
  - `system` → 系统提示词
  - `thought` → assistant 历史思考
  - `tool_call` / `tool_result` → 工具调用闭环记录
- 工具调用结束后，将内部 `messages` 局部变量归档回 `AgentContext`，而非直接销毁

### 2.2 Blackboard 分层视图

**问题**：`_build_blackboard_view` 直接截断 `raw_history`（最近 20 条），没有利用 `structured_summaries`，早期关键决策丢失。

**措施**：

- 引入"摘要-原始"双层视图：
  - 早期历史 → 由 Lead 生成 `structured_summaries` 替代
  - 近期 5-10 条 → 保留原始记录
  - `kv_state` → 结构化键值对，直接注入
- 按 Agent 角色做权限过滤（如财务 Agent 看不到用户隐私 KV）

### 2.3 AgentContext 按 Token 截断

**问题**：`trim(max_frames=20)` 按帧数截断，不同类型帧 Token 消耗差异巨大，可能导致上下文爆炸或过早丢弃关键帧。

**措施**：

- 废弃按帧数截断，引入 Token 估算（如 `tiktoken` 或模型自带 tokenizer）
- 按 Token 预算做滑动窗口，优先保留最近帧，动态丢弃早期帧

## 三、调度层改进

### 3.1 建立真实状态缓存

**问题**：`pending_agent_reports` 只被清空、从未写入，`_collect_reports` 永远返回默认假数据，LLM 调度员在做盲调。

**措施**：

- `Orchestrator` 维护 `_last_reports: Dict[str, AgentReport]`
- `_execute_assign` 返回前，将 `status_report` 写入缓存
- `_collect_reports` 优先读取真实报告，缺失时回退默认值

### 3.2 修复看门狗死锁误报

**问题**：看门狗将"所有 Agent 正常完成"误判为死锁，强制终止正常结束的会话。

**措施**：

- 死锁检测增加 `all_completed` 判断：
  - `active_count == 0 and not all_completed` → 死锁
  - `active_count == 0 and all_completed` → 正常结束
- 连续重复决策检测（同一 Agent 连续 4 轮被指派）保持现有逻辑

### 3.3 实现真并行执行

**问题**：`_execute_parallel` 实际是串行 `for` 循环，独立子任务未并发。

**措施**：

- 无依赖子任务使用 `asyncio.gather` 并发执行多个 Agent
- 结果汇总后统一写入 Blackboard，再进入调度决策
- 并发写入 Blackboard 时加版本锁或乐观锁，防止竞态

### 3.4 调度器决策输入增加工具可见性

**措施**：

- `TechLeadScheduler` 的 Prompt 中增加每个 Agent 的 `allowed_tools`
- 调度决策基于"谁能做什么"做精准指派，而非仅看角色名称

## 四、Agent 层改进

### 4.1 从贫血函数到自持实体

**问题**：`AgentConfig` 是纯数据，`AgentLoop` 是上帝类（拼消息、调 LLM、跑工具、解析状态全包），Agent 没有"活性"。

**措施**：

- 充血 Agent 模型：`Agent` 实体持有 `config`、`context`、`model`，自己负责 `think()`
- `AgentLoop` 退化为 Agent 的**内部引擎**（或并入 Agent）
- `Agent` 通过 `agent.ctx` 自主维护记忆，通过 `agent.model` 自主推理

### 4.2 Agent 作为独立协程

**问题**：Agent 是"阻塞式长事务"（`await agent_loop.run()`），主循环在 Agent 执行期间完全失明，无法中途干预、无法并行调度。

**措施**：

- 每个 Agent 是一个持续运行的 `asyncio.Task`
- 引入 `inbox/outbox` 队列与主循环异步通信：
  - `inbox`：接收调度器指令（`report_status`、`assign`、`pause`）
  - `outbox`：发送状态报告（`AgentReport`）

### 4.3 引入步进执行（Step-based）

**措施**：

- Agent 内部循环不再一次性跑完所有工具调用
- 每执行"一步"（一次 LLM 调用或一次工具调用）后：
  - 更新自身状态
  - 检查 inbox 是否有新指令
  - 让出控制权（`await asyncio.sleep(0)` 或等待 inbox）
- 调度器可以在任意步进点介入（打断、换任务、要求暂停）

### 4.4 状态机显式化

**措施**：

- Agent 内部维护显式状态机：

  ```
  IDLE → READY → RUNNING → WAITING → COMPLETED
            ↑_________|
  ```

- 主循环通过状态报告做调度，而非通过任务描述做微管理
- Agent 自主判断 `will`：`execute`（继续）、`complete`（完成）、`blocked`（阻塞）、`delegate`（建议移交）

## 五、交互层改进

### 5.1 LLM 流式输出

**问题**：LLM 调用是同步等待，用户在前端看不到 Agent 实时思考过程，体验黑箱。

**措施**：

- `ModelProvider` 增加 `chat_stream()` 接口，返回 `AsyncIterator[LLMChunk]`
- 每个 chunk 包含：`delta_text`（增量文本）、`finish_reason`、可选的 `tool_call` 指示

### 5.2 AgentLoop 双通道输出

**措施**：

- 内部缓存完整文本（用于解析 `tool_calls` 和 `status_report`）
- 同时通过 `EventBus` 实时发射事件：
  - `agent.thinking`：Agent 开始推理
  - `agent.token`：LLM 文本增量（打字机效果）
  - `agent.tool_call`：检测到工具调用意图
  - `agent.tool_result`：工具执行完毕
  - `agent.report`：状态报告（内部消费）

### 5.3 EventBus 统一总线

**目标**：调度交互全过程可观测，前端能看到"谁在指挥、为什么指挥、指挥了什么"。

**Event 结构扩展**：

```python
@dataclass
class Event:
    type: str              # "control.schedule" | "agent.token" | "user.message"
    source: str            # "orchestrator" | "agent:coder" | "watchdog"
    target: Optional[str]  # None=广播 | 定向投递
    payload: Dict
    correlation_id: Optional[str]  # 请求-响应匹配
    channel: str           # "all" | "internal" | "user"
```

**EventBus 职责**：

- 定向投递：`target` 指定收件方
- 广播：`target=None` 时推给所有订阅者
- 外部 Sink 同步：所有事件自动流向 SSE/日志/持久化

### 5.4 控制事件与观测事件分离

| 事件类型 | 示例 | 可见性 |
|---------|------|--------|
| `control.*` | `control.status_request`, `control.schedule_decision`, `control.watchdog_override` | 调度器、看门狗、前端（高级视图） |
| `agent.*` | `agent.token`, `agent.tool_call`, `agent.thought` | 前端实时流、日志 |
| `user.*` | `user.message`, `user.input_queued` | 前端、持久化 |
| `system.*` | `system.session_start`, `system.round_end` | 全部 |

### 5.5 前端过滤与体验

**用户层对话流示例**：

```
[用户] 帮我写一个爬虫
[系统] 调度员分析任务...
[系统] 调度员指派：Planner 负责设计架构
[Planner] 我来规划一下...
[Planner] 首先确定目标网站...
[系统] Planner 报告：任务分解完成，建议指派 Coder
[系统] 调度员指派：Coder 负责实现代码
[Coder] 好的，我来写代码...
```

**前端订阅策略**：

- 普通用户：只看 `channel=user` 和 `type=agent.*`
- 开发者模式：额外显示 `channel=internal` 的调度过程

## 六、工具层改进

### 6.1 按职责过滤工具可见性

**问题**：`AgentConfig.tools` 存在但未被使用，所有 Agent 都能看到全局工具池，浪费 Token 且存在安全风险。

**措施**：

- `AgentLoop` 构建工具列表时，用 `AgentConfig.tools` 对 `ToolRegistry.list_tools()` 做交集过滤
- 未声明的工具对 Agent 不可见，LLM 无法调用
- 调度器决策时参考 `allowed_tools`，精准匹配任务与能力

### 6.2 统一工具市场

**措施**：

- `ToolRegistry` 同时纳管内置工具和 MCP 工具，对外暴露统一 Schema
- MCP 只是来源之一，不是独立执行器
- 执行器单一化：一个 `ToolExecutor` 负责所有工具的实际执行，按来源路由（本地函数或 MCP 远程调用）

## 八、实施优先级

| 优先级 | 事项 | 原因 |
|--------|------|------|
| **P0** | 修复 ToolExecutor 接口断裂 | 运行时崩溃，完全不可用 |
| **P0** | 修复 `pending_agent_reports` 状态丢失 | 调度器盲调，决策无效 |
| **P0** | 修复看门狗死锁误报 | 正常任务被强制终止 |
| **P1** | 让 AgentLoop 消费 AgentContext | 每轮失忆，无法连续推理 |
| **P1** | 按 `AgentConfig.tools` 过滤工具 | Token 浪费 + 安全风险 |
| **P1** | EventSink 上提 + EventDispatcher | 事件管理混乱 |
| **P2** | LLM 流式输出 + 丰富事件类型 | 用户体验黑箱 |
| **P2** | Blackboard 分层压缩 | 长对话上下文膨胀 |
| **P3** | Agent 改造为独立协程 + 步进执行 | 架构质变，支持真并行与中途干预 |
| **P3** | EventBus 统一总线 + 控制事件可观测 | 调度过程透明化 |
| **P4** | MCP 统一接入 + Token 滑动窗口截断 | 扩展性优化 |

## 九、核心设计原则

1. **框架闭环原则**：动态决策 → 外部执行 → 基于结果再推理的流程，必须由框架维护硬循环，不能依赖 LLM 自我信号。
2. **分层控制原则**：LLM 负责语义理解和策略建议，硬代码负责边界守护和资源管理。
3. **上下文隔离原则**：全局共享事实（Blackboard），私有隔离过程（AgentContext），防止角色污染和 Token 浪费。
4. **用户体感优先原则**：底层可以多次调用、多次等待，但前端呈现为一条连贯的流式对话。
5. **全过程可观测原则**：调度交互、Agent 推理、工具调用全部通过 EventBus 透出，支持实时展示与事后审计。
