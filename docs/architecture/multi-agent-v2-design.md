# AgentHub 多智能体架构 V2 设计文档

> 本文档总结当前 AgentHub 项目向新型多智能体架构演进的设计方案与实施计划。

---

## 一、设计思想

### 1.1 核心定位

把 LLM 当作"会思考但不靠谱的团队成员"，用组织管理学的思维为其设计运行环境，用硬代码做基础设施和规则兜底，让 LLM 专注于它擅长的语义推理，而不是流程控制。

### 1.2 关键认知

- **Agent 不是被调度的资源，而是主动参与协作的实体**
- **调度员不是系统组件，而是一种角色（Team Leader）**
- **验证不是真理判定，而是分工协作的一环**
- **系统必须容许误差，在迭代中收敛**

---

## 二、总体架构：四层结构

```
┌─────────────────────────────────────────────────────────┐
│  1. 会话层（进程层）                                      │
│  - 一次完整的用户任务或对话                                │
│  - 管理生命周期（开始、暂停、结束）                         │
│  - 维护全局状态（Blackboard）                              │
├─────────────────────────────────────────────────────────┤
│  2. 智能体层（线程层）                                     │
│  - 进程内的执行单元，承载特定角色                           │
│  - 每个智能体有私有上下文（Agent Context / 栈）             │
│  - 主动报告状态，参与调度协商                               │
├─────────────────────────────────────────────────────────┤
│  3. 调度层（内核层）                                       │
│  - LLM 调度员（Meta-Agent / Team Leader）：基于语义做决策    │
│  - 硬代码看门狗（Watchdog）：执行不可逾越的硬规则            │
│  - 原则：LLM 拥有"建议权"，看门狗拥有"否决权"               │
├─────────────────────────────────────────────────────────┤
│  4. 基础设施层                                            │
│  - 网络 I/O、API 并发、流式传输缝合                         │
│  - 工具执行环境、持久化存储                                 │
│  - 支持异步并发，按业务顺序整合回会话流                       │
└─────────────────────────────────────────────────────────┘
```

## 三、核心概念

### 3.1 调度员即角色（Team Leader 模型）

调度员不是一种通用基础设施，而是**一种 Agent 角色**。不同的事务有不同的 Team Leader，各自拥有不同的提示词和调度思路。

**示例：**

| 场景 | Team Leader 角色 | 调度风格 |
|------|-----------------|---------|
| 技术方案讨论 | 技术架构师 | 收集观点 → 总结共识 → 指派执行 |
| 代码审查 | 审查组长 | 分配范围 → 收集意见 → 汇总问题 → 指派修复 |
| 危机处理 | 应急指挥官 | 快速决策 → 并行执行 → 即时反馈 → 动态调整 |

每个 Team Leader 有自己的：
- **系统提示词**（调度哲学、决策风格）
- **私有上下文**（历史决策记录、团队能力评估）
- **可调度的 Agent 池**（知道团队里有哪些人、擅长什么）

### 3.2 Agent 状态自报告

Agent 不是被动等待分配任务，而是像线程一样有自己的状态，并**主动输出状态报告**：

```json
{
  "state": "running|ready|waiting|completed",
  "will": "execute|wait|delegate|complete|blocked",
  "target_task": "coding_task_3",
  "rationale": "我发现了一个安全漏洞，建议优先修复",
  "blockers": ["等待设计师提供 UI 规范"],
  "priority": 1,
  "confidence": 0.95
}
```

**状态分层：**

| 层级 | 获取方式 | 说明 |
|------|---------|------|
| 业务状态（THINKING/TOOL_CALLING） | Agent 自报告 | 只有 Agent 自己知道 |
| 调度状态（RUNNING/SUSPENDED） | 调度器权威控制 | 防止 Agent 拒绝让出 CPU |
| 存活状态（HEARTBEAT/STALLED） | 被动探测 | 不依赖 Agent 配合 |

### 3.3 决策-执行-验证三权分立

```
调度员（Team Lead）        执行者（Worker）          验证者（Auditor）
┌─────────────┐           ┌─────────────┐          ┌─────────────┐
│ 决定：       │ ────────► │ 执行：       │ ──────► │ 验证：       │
│ - 谁来做     │           │ - 写代码     │         │ - 语法检查   │
│ - 做什么     │           │ - 写测试     │         │ - 逻辑审查   │
│ - 优先级     │           │ - 出方案     │         │ - 安全扫描   │
│ - 要不要检查 │           │             │         │             │
└─────────────┘           └─────────────┘          └──────┬──────┘
     ▲                                                    │
     └──────────────────── 反馈结果 ────────────────────────┘
```

**关键原则：**
- 调度员只决定"要不要查"，具体查什么、怎么查，由验证 Agent 负责
- 验证者也会犯错，验证误差通过多轮迭代和多验证者交叉检查来降低
- 用户可随时 override 任何决策

### 3.4 容许误差的设计哲学

多智能体系统必然存在误差：
- Agent 状态报告误差
- 调度决策误差
- 执行成果误差
- 验证误差

**不是消除误差，而是控制误差的影响范围：**
- 上下文隔离：一个 Agent 的私有上下文不会污染全局
- 成果可覆盖：下一个 Agent 可以修正上一个的错误
- 多轮收敛：第一次错了，第二次修正
- 统计优势：单个 Agent 准确率 85%，多个协作后系统整体可靠性远超 85%

## 四、数据架构：双上下文模型

### 4.1 全局上下文（Blackboard）

**定位：** 所有智能体共享的"白板"，Team Leader 负责管理内容。

**内容：**
- 原始消息历史（最近 N 条）
- 结构化摘要（由 Lead 生成）
- 键值对状态（确认事实、工具结果）

**管理策略：**
- 长期保留，但随 Token 增长进行分层压缩
- 所有智能体只读访问（或受控写入）
- 前端用户可见

**职责分层：**

| 职责 | 归属 | 说明 |
|------|------|------|
| 存储、版本控制、缓存 | BlackboardManager | 纯底层，不做语义理解 |
| 何时压缩、压缩内容 | Lead | 语义决策，理解什么重要 |
| 提取 KV 状态 | Lead | 基于语义理解提取结构化信息 |
| 构建 Agent 视图 | Lead | 不同角色看到不同粒度的信息 |

### 4.2 私有上下文（Agent Context / 栈）

**定位：** 每个智能体独享的"栈"。

**内容：**
- 系统提示词（角色身份）
- 本轮特殊指令
- ReAct 思维链
- 工具调用草稿
- 工具返回的原始数据
- 临时计算草稿

**管理策略：**
- 随轮次动态压栈/弹栈
- 推理结束后可选择性归档（用于调试）或清理（节省 Token）
- 仅当前智能体可见，前端用户默认不可见

## 五、控制流：调度与执行

### 5.1 智能体执行循环（单轮内部）

```
1. 调度器将执行权交给某智能体
2. 输入组装：私有栈 + Blackboard 只读视图（由 Lead 定制）+ 其他智能体近期产出
3. 推理：LLM 生成回复 + 状态报告
4. 工具处理：若产生工具调用，框架暂停生成，执行工具，结果压入私有栈，触发同智能体内的第二轮推理
5. 成果输出：工作产出写入共享空间，状态报告提交给调度员
```

### 5.2 会话级调度循环（轮间）

```
1. 收集所有 Agent 的状态报告
2. Team Leader 基于 Blackboard 和状态报告做出调度决策
3. 看门狗校验：
   - 同一 Agent 是否连续占用过多轮次？
   - 是否出现死锁（多 Agent 互相等待）？
   - 是否达到全局轮数/Token 上限？
4. 通过校验则执行，否则看门狗强制干预
5. 执行决策（指派/并行/等待/升级/请求用户输入）
6. 如需验证，指派验证 Agent
7. 成果归档到 Blackboard
```

### 5.3 协商式调度（非命令式）

```
不是：
  调度员 → Agent A: "你去写代码"（命令）

而是：
  调度员 → 全体成员: "当前需要实现登录功能，谁愿意承担？"
  Agent A → 调度员: "我可以，预计需要 2 轮"
  Agent B → 调度员: "我也能做，且我有相关经验"
  调度员 → Agent B: "好，由你负责，A 作为备份"
```

## 六、验证机制

### 6.1 成果验证

Agent 声称完成了某任务？**不看它说什么，看它做了什么。**

| Agent 声称 | 可验证的成果 | 验证方式 |
|-----------|------------|---------|
| "代码编写完成" | 文件系统新增/修改 | diff 检查 + 语法验证 |
| "测试编写完成" | 测试文件 + 运行结果 | pytest 执行 + 覆盖率 |
| "代码审查完成" | 审查报告 / 评论 | 报告非空 + 关键检查点 |
| "需求分析完成" | 需求文档 / 任务拆解 | 文档结构化 + 可执行性检查 |

### 6.2 分级信任

- **可验证任务**（代码、文档、测试）→ 必须成果验证
- **不可验证任务**（思考、分析、设计）→ 信任自报告，但加超时熔断

### 6.3 验证链

```
轮次 1: 程序员写代码 → 审查员说"OK" → 部署后发现 Bug
轮次 2: Bug 报告进入 Blackboard → 调度员指派修复 → 程序员修复
轮次 3: 安全员重新扫描 → 发现关联漏洞 → 再次修复
...
最终收敛到"足够好"的状态
```

## 七、死锁与冲突处理

### 7.1 死锁检测

基于 Agent 自报告的 Wait-For 图：

```
Agent A 报告: "我在等 B 的设计文档"
Agent B 报告: "我在等 A 的需求确认"
调度器: 检测到 A→B→A 循环，触发死锁消解
```

### 7.2 冲突类型与处理

| 冲突类型 | 处理方式 |
|---------|---------|
| 资源冲突 | 调度员仲裁 + 优先级继承 |
| 任务冲突 | 两阶段提交变体 |
| 知识冲突 | 乐观并发控制 + 语义合并 |
| 策略冲突 | 分布式共识 |

### 7.3 看门狗硬规则

```
- 会话最大轮数：50
- 单个 Agent 最大连续轮数：10
- 死锁检测超时：60 秒
- Token 预算：单会话上限（80% 告警）
- Agent 心跳超时：5 分钟标记为 STALLED
```

## 八、并发策略

- **API 层**：多个智能体或多次工具调用的网络请求异步并发
- **业务层**：按任务依赖关系决定
  - 独立分析（如同时查天气、股价）：并行调用多个智能体
  - 流水线任务（先规划再编码再审查）：业务串行，但每步内部子任务可并发
- **全局状态保护**：Blackboard 写入需乐观锁（version 字段）

---

## 九、单智能体到多智能体的扩展

- **单智能体模式**：调度层退化为"直通模式"，Team Leader 不介入，看门狗仅保留轮数上限和 Token 熔断。双层上下文依然保留，为后续扩展预留结构。
- **多智能体模式**：接入 Team Leader，各智能体复用相同的私有上下文规范，Blackboard 增加"智能体身份标签"和"产出时序索引"。

**核心优势**：架构无需推倒重来，只是调度策略从"固定"变为"动态"。

## 十一、关键设计原则

1. **框架闭环原则**：任何涉及"动态决策 → 外部执行 → 基于结果再推理"的流程，必须由框架维护硬循环，不能依赖 LLM 的自我信号完成闭环。

2. **分层控制原则**：LLM 负责语义理解和策略建议，硬代码负责边界守护和资源管理。

3. **上下文隔离原则**：全局共享事实（Blackboard），私有隔离过程（Agent Context），防止角色污染和 Token 浪费。

4. **用户体感优先原则**：底层可以多次调用、多次等待，但前端呈现为一条连贯的流式对话。

5. **角色即调度原则**：调度员不是系统基础设施，而是一种可替换的 Agent 角色。

6. **验证即分工原则**：验证不是真理判定，而是另一个 Agent 的分工，验证者也会犯错。

7. **误差容忍原则**：不要因为不信任而放弃多智能体带来的巨大优势，通过迭代收敛控制误差影响。

## 十二、风险与回退方案

| 风险 | 缓解措施 |
|------|---------|
| 新调度系统不稳定 | 保留旧 orchestrator，通过配置切换 |
| Blackboard 压缩策略不成熟 | 初期不压缩，只保留完整历史 |
| Agent 状态报告格式不统一 | 强制 JSON Schema，解析失败降级为默认状态 |
| Token 开销超出预期 | 增加 Token 预算熔断 |
| 前端状态管理复杂度爆炸 | 保持现有 history + streaming Map 双轨制 |

---

## 十三、最小可运行版本（MVP）

按优先级实现：

1. **P0（必须）**：Blackboard 模型 + Agent 状态报告接口
2. **P1（核心）**：一种调度策略（tech_lead）+ 看门狗基础规则
3. **P2（体验）**：前端 Agent 状态栏 + 策略选择 UI
4. **P3（优化）**：Blackboard 压缩 + 多策略切换 + 验证 Agent 体系

预计 **4-6 周**可运行 MVP，现有功能不受影响。
