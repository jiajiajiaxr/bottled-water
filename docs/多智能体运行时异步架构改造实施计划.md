# 多智能体运行时异步架构改造实施计划

## 背景与目标

当前后端运行时采用**同步轮询式调度**：Orchestrator 做主循环，每轮"收集报告 → 调度器决策 → 阻塞执行 Agent → 等待完成 → 下一轮"。调度器（TechLeadScheduler）是一个策略类，不是平等的 Agent；Agent 执行是阻塞长事务，执行期间调度器完全失明。

目标是将架构改造为**异步事件驱动多智能体系统**：所有 Agent（含调度员）是并发的 `asyncio.Task`，通过 EventBus 和共享 BlackBoard 协作。调度员持续监视全局状态，随时发布指令或质询；工作 Agent 步进执行，每步之间检查 inbox 并汇报状态。

> 本计划对应 `docs/多智能体运行时架构改进方案.md` 中的 **P3-P4** 改造项。

---

## 当前架构 vs 目标架构

### 当前（同步轮询）

```
Orchestrator._scheduling_loop()
  while running:
    reports = _collect_reports()
    decision = await scheduler.make_decision()   // 阻塞等 LLM
    await _execute_decision(decision)            // 阻塞等 Agent 跑完
    // Agent 执行期间调度器完全不参与
```

### 目标（异步事件驱动）

```
Orchestrator 启动所有 AgentActor（含调度员）as asyncio.Task

调度 AgentActor                工作 AgentActor A             工作 AgentActor B
      |                                |                             |
      | 订阅: agent.report             |  inbox: control.assign      |  inbox: control.assign
      | 订阅: blackboard.updated       |                              |
      |                                |                              |
      |←────────── agent.report ───────┤                              |
      |←────────── agent.report ─────────────────────────────────────┤
      |                                |                              |
      ├────── control.assign ───────→ |                              |
      |                                |  step() 执行一步              |
      |                                |  发布 agent.report           |
      |←────────── agent.report ───────┤                              |
      |                                |                              |
      ├────── control.assign ─────────────────────────────────────→  |
      |                                                              |  step()
      |←────────────────────────────────────── agent.report ─────────┤
```

---

## 核心设计决策

### 1. EventBus 通信协议

扩展现有 `EventDispatcher`，增加**发布/订阅**能力（当前只有 dispatch-to-sink）。

**定向通信事件：**
- `control.assign` — 调度员指派任务给某 Agent（target=agent_id）
- `control.pause` — 调度员暂停某 Agent
- `control.resume` — 调度员恢复某 Agent
- `agent.report` — Agent 汇报当前状态（target=orchestrator 或广播）
- `agent.token` / `agent.tool_call` / `agent.tool_result` — 观测事件（广播）
- `blackboard.updated` — BlackBoard 内容变更（广播）

**Event 字段复用：**
- `Event.source` — `"scheduler"` / `"agent:coder"` / `"system"`
- `Event.target` — `None`（广播）或 `agent_id`（定向）
- `Event.channel` — 保留现有 `"all"` / `"internal"` / `"user"`

### 2. Agent 步进粒度

将 `AgentLoop._execute_loop` 拆分为 3 个步骤：

- **Step 1: `think`** — 构建 prompt，调 LLM，获取 response（含可能的 tool_calls）
- **Step 2: `act`** — 如果有 tool_calls，执行第一个工具调用（支持多轮 tool 时循环 step 2）
- **Step 3: `report`** — 解析最终回复，提取 status_report，发布 `agent.report`

每步之间：
1. 更新自身状态机
2. 检查 inbox 是否有新指令（`control.pause` / `control.assign`）
3. 让出控制权（`await asyncio.sleep(0)`）

### 3. 调度员 Agent 化

`TechLeadScheduler` → `SchedulerAgent`（继承 `AgentActor`）：

- **Prompt 改造**：从"现在请你做调度决策"改为"你是一个调度员，持续监视全局状态，为各 Agent 分配任务。当你觉得需要干预时，输出指令。"
- **输入来源**：订阅 `agent.report` 和 `blackboard.updated`，而非每轮主动收集
- **输出形式**：发布 `control.assign` / `control.pause` 等事件到 EventBus
- **自身也是 Agent**：有自己的 `AgentContext`，可以积累调度经验

### 4. Orchestrator 角色退化

从"主循环调度者"退化为"生命周期管理 + 任务监控"：

- 初始化：创建所有 AgentActor（含调度员），启动 `asyncio.Task`
- 运行时：不再做 while 循环，只监控健康状态（看门狗）
- 清理：会话结束时取消所有 Agent Task

## 实施阶段

### Phase 1: EventBus 基础设施（1 天）

**目标**：让 EventDispatcher 支持 Agent 间的发布/订阅通信。

**任务：**

1. **扩展 `EventDispatcher`**（`src/agent_runtime/runtime/event_dispatcher.py`）
   - 增加 `subscribe(event_type: str, callback: Callable)` 和 `unsubscribe()`
   - 增加 `publish(event: Event)` 方法（定向投递 + 广播）
   - 保持现有的 `dispatch()` 不变（前端 Sink 输出）
   - 事件按 `target` 字段路由：有 target 则只投递给该订阅者，无 target 则广播

2. **引入 `inbox/outbox` 机制**
   - 新建 `src/agent_runtime/runtime/mailbox.py`
   - `Mailbox` 类：每个 Agent 一个 inbox（`asyncio.Queue[Event]`），Orchestrator 一个控制 inbox
   - 提供 `send(target, event)` 和 `recv() -> Event` 接口

3. **定义通信协议常量**
   - 新建 `src/agent_runtime/core/protocol.py`
   - 定义所有事件类型字符串常量：`CONTROL_ASSIGN`、`CONTROL_PAUSE`、`AGENT_REPORT`、`BLACKBOARD_UPDATED` 等

**验收标准：**
- 单元测试：两个协程通过 EventBus 互相发送/接收事件，验证定向投递和广播都正确
- 不改动任何现有业务逻辑

---

### Phase 2: AgentLoop 步进化（1.5 天）

**目标**：将 `AgentLoop` 从"一次性跑完"改造为"步进执行"。

**任务：**

1. **拆分 `_execute_loop` 为 `step()` 序列**
   - 新建 `src/agent_runtime/runtime/agent_stepper.py`（或复用 `agent_loop.py`）
   - 将内部逻辑拆分为 `_step_think()`、`_step_act()`、`_step_report()`
   - 每个 step 返回 `(next_step_name: str, events: List[Event])`
   - 状态机：`IDLE → THINK → ACT → REPORT → COMPLETED`

2. **引入外部干预点**
   - `AgentLoop` 增加 `inbox: asyncio.Queue[Event]` 参数
   - 每步之间调用 `_check_inbox()`：
     - 收到 `control.pause` → 状态改为 `PAUSED`，等待 `control.resume`
     - 收到 `control.assign` → 更新 `current_task`，重新从 THINK 开始
   - 非阻塞检查：`try: inbox.get_nowait() except QueueEmpty: pass`

3. **保持 `run()` 向后兼容**
   - `run()` 内部变成 `while step_name != "completed": step()` 的包装
   - 现有调用方（Orchestrator._execute_assign）不需要立即改动

**验收标准：**
- 单元测试：AgentLoop 在 step 之间收到 `control.pause` 会暂停，收到 `control.assign` 会切换任务
- `_execute_assign` 仍然可以 `await agent_loop.run()` 正常工作

---

### Phase 3: AgentActor 引入（1.5 天）

**目标**：每个 Agent 是一个独立的 `asyncio.Task`。

**任务：**

1. **新建 `AgentActor` 类**（`src/agent_runtime/runtime/agent_actor.py`）
   - 属性：`agent_id`、`config`、`agent_loop`、`inbox: Queue`、`state: AgentState`
   - 方法：
     - `async def run()` — 主协程循环：从 inbox 等指令 → 执行 step → 发布事件 → 循环
     - `async def handle_event(event)` — 处理外部事件（assign/pause/resume）
     - `def get_status() -> AgentReport`

2. **状态机显式化**
   - `IDLE → READY → RUNNING → PAUSED → WAITING → COMPLETED`
   - 状态转换通过 `handle_event` 触发
   - 状态变更时发布 `agent.state_changed` 事件

3. **BlackBoard 事件通知**
   - 扩展 `BlackboardManager`：每次 `append_history` / `update_kv` 后，发布 `blackboard.updated` 事件到 EventBus
   - 调度 Agent 订阅此事件

**验收标准：**
- 单元测试：启动 3 个 AgentActor，通过 EventBus 互相发送事件，验证各自状态正确

---

### Phase 4: 调度员 Agent 化（2 天）

**目标**：将 `TechLeadScheduler` 改造为 `SchedulerAgent`。

**任务：**

1. **新建 `SchedulerAgent` 类**（`src/agent_runtime/strategies/scheduler_agent.py`）
   - 继承 `AgentActor`
   - 复用 `TechLeadScheduler` 的 LLM prompt 逻辑，但改造为"持续监视"视角

2. **Prompt 改造**
   - 新 Prompt："你是一个调度员，负责协调团队成员。你可以随时输出指令。当前各成员状态：{agent_reports}。BlackBoard 最新摘要：{blackboard_summary}。请决定是否需要发布指令。"
   - LLM 输出格式：JSON，包含 `action`（`assign`/`pause`/`wait`/`complete`）和 `targets`（目标 Agent 列表）

3. **事件订阅与发布**
   - `SchedulerAgent.run()` 中订阅 `agent.report` 和 `blackboard.updated`
   - 收到事件后触发 `think()`（调 LLM）
   - LLM 输出指令后，通过 EventBus 发布 `control.*` 事件到对应 Agent 的 inbox

4. **处理 `workflow` 模式**
   - `WorkflowScheduler` 保持不变（图遍历逻辑不需要 LLM）
   - 但 `WorkflowScheduler` 也需要接入 EventBus：遍历到下一个节点时发布 `control.assign`
   - 或者为 Workflow 模式写一个轻量的 `WorkflowSchedulerAgent`

**验收标准：**
- 集成测试：2 个工作 Agent + 1 个调度 Agent，模拟用户任务，验证调度员能正确指派和协调
- 调度员自己的消息也出现在前端消息流中

---

### Phase 5: Orchestrator 重构（1.5 天）

**目标**：从同步轮询改为事件驱动的生命周期管理。

**任务：**

1. **重构 `Orchestrator`**（`src/agent_runtime/runtime/orchestrator.py`）
   - 删除 `_scheduling_loop()` 中的 while 循环
   - `run()` 方法改为：
     1. 初始化所有 AgentActor（工作 Agent + 调度 Agent）
     2. 启动所有 `asyncio.Task`
     3. 启动事件收集协程：从 EventBus 收集所有事件并 `yield`
     4. 等待 `session_completed` 信号（`asyncio.Event`）
     5. 取消所有 Agent Task，清理资源

2. **引入 `session_completed` 信号**
   - 当调度 Agent 发布 `control.complete` 或所有 Agent 都 `COMPLETED` 时，触发 `session_completed.set()`
   - 看门狗也监听此信号，超时则强制终止

3. **保持 `EventDispatcher` 集成**
   - AgentActor 产生的事件通过 EventBus 广播
   - Orchestrator 收集事件并 `yield`，同时通过 `EventDispatcher.dispatch()` 推送到前端 Sink

**验收标准：**
- 端到端测试：完整走一遍"用户发送消息 → 调度员制定计划 → 工作 Agent 执行 → 返回结果"流程
- WebSocket 前端能正常接收所有事件

---

### Phase 6: 集成测试与前端适配（1 天）

**目标**：确保改造后系统稳定，前端展示正常。

**任务：**

1. **现有测试修复**
   - `test_tech_lead_scheduler.py` — 适配 SchedulerAgent 的新接口
   - `test_orchestrator.py` — 适配事件驱动架构

2. **新增测试**
   - `test_agent_actor.py` — AgentActor 状态机、inbox 处理
   - `test_eventbus.py` — 定向投递、广播、订阅/取消订阅
   - `test_scheduler_agent.py` — 调度员 Agent 的协调逻辑

3. **前端事件适配**
   - 新增事件类型：`control.assign`、`control.pause`、`agent.state_changed`
   - 前端需要能展示调度员的"指令"消息（作为系统消息或特殊样式）
   - 保持现有 `agent.token` / `agent.tool_call` 等事件的兼容性

4. **回归测试**
   - 单 Agent 模式（`SingleAgentScheduler`）不受影响
   - Workflow 模式正常工作

**验收标准：**
- 所有后端测试通过
- 前端聊天流程正常（发送消息 → 接收回复 → 展示工具调用）
- 多 Agent 群聊能正确展示多个 Agent 的并行输出

## 风险与对策

| 风险 | 影响 | 对策 |
|------|------|------|
| 事件时序混乱 | 调度员在 Agent 还没汇报时就发了新指令 | 引入 `agent.report` 的 `correlation_id`，调度员只处理最新报告；EventBus 保证同一会话内事件按产生顺序投递 |
| LLM 调用并发过多 | 多个 Agent 同时调 LLM，Token 费用爆炸 | 在 `ModelProvider` 层引入并发令牌桶（semaphore），限制同时调用的 Agent 数量；调度员可以 `control.pause` 暂停低优先级 Agent |
| 死循环 | 调度员反复指派同一个 Agent | 看门狗已存在的"连续重复决策检测"仍然有效；SchedulerAgent 的 prompt 中加入"避免重复指派同一 Agent"的约束 |
| 前端事件乱序 | 异步后事件可能不按预期顺序到达 | Event 增加 `sequence_number`；前端按 sequence 排序展示 |
| 状态不一致 | Agent 认为自己在 RUNNING，调度员认为它 IDLE | 以 `agent.report` 为唯一真相来源；调度员不信任自己的缓存，每次都等最新报告 |

## 建议的实施顺序

考虑到风险控制和可回滚性，建议按以下顺序执行：

1. **先做 Phase 1 + Phase 2**（EventBus + 步进执行）：这是所有后续改造的基础设施，且可以独立测试
2. **review 后继续 Phase 3 + Phase 4**（AgentActor + 调度员 Agent 化）：核心架构改造
3. **最后 Phase 5 + Phase 6**（Orchestrator 重构 + 集成测试）：收尾

每个阶段完成后都可以单独提交并通过测试，不要一次性合并所有改动。
