# 多智能体聊天应用架构设计总结

## 一、总体分层：四层结构

### 1. 会话层（进程层）
- **定位**：一次完整的用户任务或对话，对应操作系统中的"进程"。
- **职责**：管理生命周期（开始、暂停、结束）、维护全局状态、隔离不同用户会话。
- **核心资产**：全局上下文（Blackboard），即所有智能体共享的"堆"。

### 2. 智能体层（线程层）
- **定位**：进程内的执行单元，对应"线程"。
- **职责**：承载特定角色（如规划师、程序员、审查员），执行推理、生成回复、调用工具。
- **核心资产**：私有上下文（Agent Context），即每个智能体独享的"栈"，包含角色设定、思维链、工具中间结果和临时草稿。

### 3. 调度层（内核层）
- **定位**：决定哪个智能体在何时获得执行权。
- **组成**：
  - **LLM 调度员（Meta-Agent）**：基于语义理解做复杂仲裁，处理"该谁说话"的决策。
  - **硬代码看门狗（Watchdog）**：执行不可逾越的硬规则，如轮数上限、防死锁、防重复、Rate Limit 保护。
- **原则**：LLM 调度员拥有"建议权"，看门狗拥有"否决权"。

### 4. 基础设施层
- **职责**：管理网络 I/O、API 并发、流式传输缝合、工具执行环境、持久化存储。
- **关键特性**：支持异步并发调用大模型，但将并发结果按业务顺序整合回会话流。

---

## 二、数据架构：双上下文模型

### 全局上下文（共享堆）
- **内容**：用户消息历史、已确认的事实摘要、共享知识库、持久化记忆、其他智能体的最终产出。
- **管理策略**：长期保留，但随 Token 增长进行分层压缩（原始历史 → 结构化摘要 → 键值对状态）。
- **可见性**：所有智能体只读访问（或受控写入），前端用户可见。

### 私有上下文（私有栈）
- **内容**：系统提示词（角色身份）、本轮特殊指令、ReAct 思维链、工具调用草稿、工具返回的原始数据、临时计算草稿。
- **管理策略**：随轮次动态压栈/弹栈，推理结束后可选择性归档（用于调试）或清理（节省 Token）。
- **可见性**：仅当前智能体可见，前端用户默认不可见。

---

## 三、控制流：调度与执行

### 1. 智能体执行循环（单轮内部）
- **触发**：调度器将执行权交给某智能体。
- **输入组装**：该智能体的私有栈 + 全局堆的只读视图 + 其他智能体的近期产出摘要。
- **推理**：LLM 生成回复，可能包含文本、结构化决策或工具调用信号。
- **工具处理**：若产生工具调用，框架立即暂停生成，执行工具函数，将结果压入该智能体的私有栈，触发**同智能体内的第二轮推理**（这是由框架维护的硬循环，非 LLM 自驱）。
- **输出归档**：最终答案写入全局上下文，私有栈的中间过程可选择性丢弃。

### 2. 会话级调度循环（轮间）
- **触发**：每轮结束后，调度层重新评估。
- **LLM 调度员**：收集各智能体的"意愿报告"（如"我有重要发现"或"我暂时无事"），结合当前任务阶段，提议下一个执行者。
- **看门狗校验**：
  - 同一智能体是否已连续占用过多轮次？
  - 是否出现死锁（多智能体互相谦让）或活锁（多智能体争抢）？
  - 是否达到全局轮数上限？
- **决策**：通过校验则执行，否则强制干预（如指定默认智能体、强制总结、或终止会话）。

---

## 四、工具调用机制

- **本质**：工具调用是 LLM 的"文本输出"被框架解析后触发的副作用，LLM 本身不执行任何外部代码。
- **因果约束**：由于 LLM 是因果模型，生成工具调用参数时不可能预知工具返回结果，因此**至少需要两次 LLM 调用**（一次决策，一次总结）。
- **前端体验**：基础设施层可将两次调用的流式输出拼接为一条连续的 SSE 流，使用户体感上觉得"边想边做"，但底层仍是两次独立请求。
- **优化路径**：若工具参数可预计算（不依赖 LLM 推理），则可在调用 LLM 前先执行工具，将结果直接注入上下文，实现真正的单次调用。

---

## 五、并发策略

- **API 层**：多个智能体或多次工具调用的网络请求应异步并发，充分利用 I/O 等待时间。
- **业务层**：是否并发取决于任务依赖关系：
  - **独立分析**（如同时查天气、股价、汇率）：可并行调用多个智能体。
  - **流水线任务**（如先规划再编码再审查）：业务上串行，但每步内部的子任务可并发。
- **全局状态保护**：并发写入共享 Blackboard 时需加锁或版本控制，防止竞态。

---

## 六、单智能体到多智能体的扩展路径

- **单智能体模式**：调度层退化为"直通模式"，看门狗仅保留轮数上限和 Token 熔断。双层上下文依然保留，为后续扩展预留结构。
- **多智能体模式**：直接接入 LLM 调度员，各智能体复用相同的私有上下文规范，全局上下文增加"智能体身份标签"和"产出时序索引"。
- **核心优势**：架构无需推倒重来，只是调度策略从"固定"变为"动态"。

---

## 七、关键设计原则

1. **框架闭环原则**：任何涉及"动态决策 → 外部执行 → 基于结果再推理"的流程，必须由框架维护硬循环，不能依赖 LLM 的自我信号完成闭环。
2. **分层控制原则**：LLM 负责语义理解和策略建议，硬代码负责边界守护和资源管理。
3. **上下文隔离原则**：全局共享事实，私有隔离过程，防止角色污染和 Token 浪费。
4. **用户体感优先原则**：底层可以多次调用、多次等待，但前端呈现为一条连贯的流式对话。

---

> **核心思想**：把 LLM 当作"会思考但不靠谱的团队成员"，用操作系统进程-线程的思维为其设计运行环境，用硬代码做基础设施和规则兜底，让 LLM 专注于它擅长的语义推理，而不是流程控制。
